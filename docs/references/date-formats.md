# Genealogical Date Formats

> Parsing, formatting, validation, and all date logic are handled by the in-app module `@vata-apps/gedcom-date`. This document covers what matters for the Vata app: user input formats, storage, and display. See [ADR-004](../decisions/adr-004-gedcom-libraries.md).

---

## User Input

The `DateInput` component accepts the following formats. The input is case-insensitive.

| What the user types | Meaning               |
| ------------------- | --------------------- |
| `15 JAN 1845`       | Exact date            |
| `JAN 1845`          | Month and year only   |
| `1845`              | Year only             |
| `1845-01-15`        | ISO format            |
| `15/01/1845`        | French notation       |
| `ABT 1845`          | About 1845            |
| `CAL 1845`          | Calculated to be 1845 |
| `EST 1845`          | Estimated 1845        |
| `BEF 1845`          | Before 1845           |
| `AFT 1845`          | After 1845            |
| `BET 1840 AND 1845` | Between 1840 and 1845 |
| `FROM 1840 TO 1845` | From 1840 to 1845     |

The `DateInput` shows a real-time preview of how the date will be interpreted and displays a validation error if the format is unrecognized.

## Database Storage

Each date is stored as two columns in the `events` table:

| Column          | Type   | Content                              | Example           |
| --------------- | ------ | ------------------------------------ | ----------------- |
| `date_original` | `TEXT` | Exact text entered by the user       | `ABT 15 JAN 1845` |
| `date_sort`     | `TEXT` | ISO string for chronological sorting | `1845-01-15`      |

- `date_original` is always preserved as-is. This is the source of truth.
- `date_sort` is generated by `toSortDate(parse(dateOriginal))` at creation/update time.
- Missing month or day defaults to `01` in the sort date.
- Ranges use the start date for sorting.

## UI Display

### Date Formats

Three display formats are available, chosen depending on the context:

| Context                       | Format | Example                  |
| ----------------------------- | ------ | ------------------------ |
| Tables, lists, compact spaces | short  | `~15/01/1845`            |
| Default display               | medium | `about 15 Jan 1845`      |
| Detail views, headers         | long   | `about January 15, 1845` |

Display is locale-aware (English by default, French when the app locale is `fr`).

### Lifespan (PersonCard, lists)

Compact birth-death display used on cards and in individual lists:

| Situation             | Display      |
| --------------------- | ------------ |
| Known birth and death | `1845-1920`  |
| Approximate birth     | `~1845-1920` |
| Still living          | `1845-`      |
| Unknown birth         | `?-1920`     |

### Age at Death / at Event

| Situation                     | Display     |
| ----------------------------- | ----------- |
| Exact dates                   | `75 years`  |
| At least one approximate date | `~75 years` |

Displayed on the person header and in event details.

## Best Practices for Data Entry

1. **Use modifiers** -- prefer `ABT 1845` over guessing an exact date
2. **Be precise** -- use `BEF 1845` rather than `ABT 1844`
3. **Use intervals for uncertainty** -- `BET 1840 AND 1850` when you cannot narrow it down
4. **Cite calculated dates** -- a `CAL` date should have a source citation explaining the calculation
